Portfolio Aggregator Specification
==================================

Purpose
=======

The aggregator normalizes holdings exported from supported institutions and manually
maintained real estate into one self-contained, versioned portfolio JSON document.
The JSON document is the only portable output consumed by Portfolio Explorer.


Command-Line Interface
======================

Exactly one existing portfolio directory is required:

    python aggregate.py portfolio-sample

The default output is <portfolio-directory>/portfolio.json. An existing file is
atomically replaced. --output may specify a different filename in the portfolio
directory, but the filename must use the .json extension.


Portfolio Directory Layout
==========================

A runnable portfolio uses this layout:

    <portfolio-directory>/
      inputs/
        config.json
        <holdings CSV files>
      cache/
        fx.csv
        yields.csv
      portfolio.json

inputs contains user-maintained source data and configuration. The aggregator reads
but never modifies inputs. Holdings discovery is non-recursive over inputs/*.csv;
every discovered CSV must match exactly one registered parser.

cache contains aggregator-managed, reproducible inputs. The aggregator creates the
directory and missing cache files when necessary. Deleting a cache file explicitly
requests a refresh on the next run.

portfolio.json is the sole generated output. It must contain enough configuration,
rates, yield provenance, metadata, and holdings to be used independently of inputs
and cache.


Configuration
=============

inputs/config.json contains these keys:

- account_columns: ordered output account names.
- real_estate_account: account used by real-estates.csv.
- allowed_asset_types, allowed_markets, and allowed_risks.
- wealthsimple_account_types: provider account type to account-column mappings.
- td_direct_investing_accounts: account number to account column and currency.
- assets: Type/Market/Sector/Risk metadata keyed by market asset identifier.

The complete validated configuration document is embedded unchanged in
portfolio.json under configuration.


Input Parsers
=============

Each institution or manual format implements the common parser interface. The sample
portfolio covers Wealthsimple, TD Direct Investing, and real-estates.csv.

Real estate uses this format:

    Name,Market,Risk,Currency,Value,Net Monthly Income,URL

Value must be positive. Yield is calculated in percentage points as:

    Net Monthly Income * 12 / Value * 100

Real estate Type and Sector are Real Estate. Market, Risk, Currency, and optional
absolute HTTP/HTTPS URL come from the row.


JSON Output Contract
====================

portfolio.json has this top-level shape:

    {
      "schemaVersion": 1,
      "configuration": { ...complete config.json... },
      "exchangeRates": { ... },
      "yields": { ... },
      "supplementalAssets": { ... },
      "holdings": [ ... ]
    }

schemaVersion is required and currently equals 1.

exchangeRates is keyed by currency. Each value contains:

- rateToCad: positive numeric conversion rate.
- date: ISO observation date or null.
- source: provenance text or null.

yields is keyed by asset. Each value contains:

- providerSymbol: provider identifier or null.
- percent: yield in percentage points or null when unavailable.
- asOf: ISO date or null.
- source: provenance text or null.
- status: cache or derivation status.

Market-yield records reproduce the complete yields cache, including cached assets not
currently held. CASH-CAD and CASH-USD receive fixed zero-yield records. Real-estate
yields receive derived records.

supplementalAssets contains metadata only for assets not represented in
configuration.assets. It therefore contains built-in cash assets and real estate.
Each record contains type, market, sector, and risk. Real estate may additionally
contain url. Market URLs are not duplicated: Explorer derives them from the yield
record's providerSymbol.

holdings contains one record per asset:

    {
      "asset": "MSFT",
      "currency": "USD",
      "accounts": {
        "RRSP": 15000,
        "TFSA": 8000
      }
    }

Only nonzero account values are emitted. Missing configured accounts mean zero.
Holding records do not duplicate Type, Market, Sector, Risk, yield, URL, or exchange
rate information.

The output deliberately excludes Total, Total CAD, % Holding, and Projected Annual
Income. These are presentation-dependent derived values calculated by Explorer from
the selected accounts.


Cash Holdings
=============

CASH-CAD and CASH-USD holdings always exist, even when all account values are zero.
Their metadata is built in:

- Type: Cash
- Sector: Cash
- Risk: Low
- Yield: 0
- CASH-CAD: Canada and CAD
- CASH-USD: US and USD


Exchange-Rate Cache
===================

cache/fx.csv uses:

    Currency,Rate CAD,Date,Source

CAD must exist with rate 1 and USD must exist. Codes are uppercase, rates are
positive Decimals, duplicates are invalid, and dates use ISO YYYY-MM-DD.

If cache/fx.csv is missing, the latest FXUSDCAD observation is fetched from the Bank
of Canada Valet API and the complete cache is written atomically. Existing cache
files are always reused.


Yield Cache
===========

cache/yields.csv uses:

    Asset,Provider Symbol,Yield,As Of,Source,Status

Stocks and ETFs missing from the cache are fetched through yfinance. The annualized
latest-distribution yield is:

    latest positive distribution
    * number of positive distributions in the previous year
    / latest closing price
    * 100

No distributions means zero yield. Provider failure means a null output yield and
unavailable status, not zero. Existing rows are authoritative. A row with refresh
status is fetched again using its cached provider symbol. Writes are atomic.


Validation
==========

The aggregator fails clearly for missing directories/configuration, ambiguous parser
matches, unmapped accounts or assets, inconsistent asset metadata/currency/yield/URL,
invalid numbers or URLs, missing FX currencies, malformed cache files, and provider
request/response failures. A failed managed-file write must not leave a partial file.
